# AI 编写代码规范

## 1. 目标与适用范围

本规范用于指导 AI 或开发者在轻量级 Python 项目中稳定地产出可读、可运行、可维护的代码。

适用场景：

- 单文件或少量模块组成的桌面工具、脚本、小型服务
- 依赖数量较少、发布流程简单的 Python 项目
- 需要快速迭代，但仍希望保持基本工程质量的仓库

本规范强调以下原则：

- 优先简单：先选择直接、清晰、低依赖的实现
- 优先可读：代码应让下一位维护者快速理解
- 优先稳定：避免为了“聪明”而引入不必要复杂度
- 优先局部改动：修改范围尽量小，减少回归风险

## 2. 项目结构建议

轻量级 Python 项目建议保持扁平结构，但要避免所有逻辑无限堆积到单个文件。

推荐目录：

```text
project-root/
  docs/
  tests/
  main.py
  app.py
  utils.py
  pyproject.toml
  requirements.txt
  README.md
```

当业务逐渐增长时，再按职责拆分，例如：

- `api/`：网络请求、数据拉取
- `ui/`：界面逻辑
- `services/`：业务编排
- `utils/`：纯工具函数
- `types/`：类型定义或协议

规范要求：

- 单个文件过大时优先拆分，建议超过 300 到 500 行开始评估
- 模块名使用小写下划线风格
- 避免建立过深目录层级，轻量项目控制在 2 到 3 层内

## 3. 编码总原则

### 3.1 可读性优先

- 函数应表达单一职责
- 变量命名应体现真实含义，不使用模糊缩写
- 控制逻辑优先写成线性流程，少用过度嵌套
- 同一文件内的代码顺序保持稳定：常量、工具函数、核心类、入口函数

### 3.2 简单优先

- 能用标准库解决，就不要优先引入第三方依赖
- 能用明确的 `if/else` 解决，就不要过早抽象
- 不为未来假设的需求设计复杂扩展点

### 3.3 小步修改

- 每次改动只处理一个明确问题
- 修复 bug 时不要顺手重构无关模块
- 如果必须重构，应将“行为变更”和“结构整理”分开提交

## 4. 命名规范

统一采用 Python 社区常见约定：

- 变量与函数：`snake_case`
- 类名：`PascalCase`
- 常量：`UPPER_SNAKE_CASE`
- 私有辅助方法：前缀 `_`

命名要求：

- 名称应说明用途，而不是说明类型
- 布尔值建议使用可判断语义，例如 `is_visible`、`has_error`
- 集合变量使用复数语义，例如 `records`、`items`
- 避免单字符变量，循环计数器等极短局部变量除外

示例：

```python
MAX_RETRY_COUNT = 3

def load_price_data(source_url: str) -> dict:
    ...

class PriceCalculator:
    ...
```

## 5. 函数与类设计规范

### 5.1 函数规范

- 单个函数建议控制在 20 到 50 行内
- 入参数量尽量不超过 5 个，过多时考虑封装对象
- 函数名使用动词短语，如 `load_data`、`calculate_average`
- 返回值应稳定，避免同一函数有时返回 `dict`，有时返回 `str`

优先推荐：

- 小函数
- 纯函数
- 明确输入输出

避免：

- 超长函数
- 隐式修改外部状态
- 一个函数同时做校验、计算、存储、渲染

### 5.2 类规范

- 只有在状态管理或职责聚合明显受益时才引入类
- 工具函数集合不要强行包装为类
- 类的方法应围绕同一业务对象展开

轻量项目中优先选择：

- 简单函数 + 少量清晰类

而不是：

- 大而全的管理器类
- 多层继承
- 复杂设计模式堆叠

## 6. 类型标注规范

轻量级项目也应尽可能补充类型标注，尤其是公共函数与关键业务函数。

要求：

- 新增函数默认添加参数与返回值类型
- 复杂结构优先使用 `TypedDict`、`dataclass` 或清晰注释说明
- 对可空值显式使用 `Optional[...]`

示例：

```python
from typing import Optional


def parse_price(text: str) -> Optional[float]:
    ...
```

如果项目还未全面启用类型检查，也应先从新增代码开始执行。

## 7. 异常处理规范

异常处理要明确、可定位、不过度吞错。

要求：

- 仅捕获预期异常
- 捕获后给出明确上下文信息
- 除非已完成降级处理，否则不要静默忽略异常

推荐写法：

```python
try:
    response = requests.get(url, timeout=5)
    response.raise_for_status()
except requests.RequestException as exc:
    raise RuntimeError(f"获取金价数据失败: {url}") from exc
```

避免写法：

```python
try:
    ...
except Exception:
    pass
```

## 8. 日志与输出规范

轻量项目允许使用简单输出，但要区分“调试信息”和“用户信息”。

建议：

- 调试阶段可使用 `print`
- 稍长期维护的项目优先切换到 `logging`
- 错误信息应包含动作、对象、原因

例如：

- 好：`加载配置失败：config.json 不存在`
- 差：`error`

## 9. 注释与文档规范

注释应解释“为什么”，而不是重复“代码做了什么”。

要求：

- 复杂分支前允许添加 1 到 2 行说明
- 公共函数在必要时写简短 docstring
- 文件顶部可添加模块用途说明

避免：

- 大量显而易见的逐行注释
- 注释与代码长期不一致

## 10. 依赖管理规范

轻量级 Python 项目要严格控制依赖数量。

要求：

- 引入第三方库前先确认标准库无法满足
- 每个依赖都应有明确用途
- 能删除的依赖及时删除

建议：

- 使用 `requirements.txt` 或 `pyproject.toml` 统一管理依赖
- 新增依赖时同步更新 README 中的安装说明

## 11. 配置与常量规范

- 可变配置不要硬编码在业务函数内部
- 路径、接口地址、刷新间隔等值应提升为模块常量或配置项
- 默认值应集中定义，避免散落在多个函数中

示例：

```python
DEFAULT_REFRESH_INTERVAL = 0.5
DEFAULT_TIMEOUT_SECONDS = 5
```

## 12. I/O 与外部调用规范

涉及文件、网络、系统调用时，应优先保证可恢复性和可测试性。

要求：

- 网络请求必须设置超时
- 文件读写明确编码，默认使用 `utf-8`
- 外部接口结果先校验，再进入业务逻辑

## 13. 测试规范

轻量项目不要求一开始就建立完整测试体系，但新增核心逻辑时应逐步补上最小测试覆盖。

优先测试以下内容：

- 纯计算函数
- 输入校验逻辑
- 异常分支
- 数据转换逻辑

建议：

- 测试文件放在 `tests/` 目录
- 文件名使用 `test_*.py`
- 每个测试只验证一个明确行为

示例：

```python
def test_calculate_average_price():
    result = calculate_average_price(10, 100, 20, 200)
    assert result == 15
```

## 14. AI 编码行为约束

当由 AI 生成或修改代码时，必须遵守以下限制：

- 不无端重命名已有稳定接口
- 不擅自引入大型框架
- 不一次性改动多个无关文件
- 不为了“更优雅”而替换成熟可用的简单实现
- 不覆盖用户已有的手工修改，除非任务明确要求

AI 输出代码前应自检：

- 是否符合当前仓库风格
- 是否引入了不必要依赖
- 是否改变了已有行为
- 是否缺少异常处理
- 是否需要补充文档或测试

## 15. 提交前检查清单

每次提交前，至少检查以下项目：

- 代码可以运行
- 导入未失效
- 未留下调试输出或临时代码
- 命名清晰
- 异常处理合理
- 文档已同步更新

## 16. 对本仓库的落地建议

结合当前仓库形态，建议优先执行以下规则：

- 保持核心脚本简洁，避免过早拆成复杂包结构
- 将界面逻辑、计算逻辑、构建逻辑逐步分离
- 对网络请求和价格解析补充类型标注与异常处理
- 后续新增模块时统一纳入 `docs/` 文档说明

## 17. 最小执行标准

如果时间有限，至少满足以下最低标准：

1. 新增函数补齐类型标注
2. 网络请求设置超时并处理异常
3. 避免超长函数和模糊命名
4. 新增依赖时同步更新说明文档
5. 改动核心逻辑时补充最小测试或示例

以上标准适用于大多数轻量级 Python 项目，也适合作为 AI 参与开发时的默认约束。
